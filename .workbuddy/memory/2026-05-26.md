## FrostedAnimatedGlowView HW 路径优化
- 实现了真正的 HW 路径（API 31+）：dispatchDraw 中使用 RenderNode + RenderEffect 替代 mDisplayBitmap 绘制
- `captureAndBlurFromSurfaceView`：API 31+ 上 PixelCopy 完成后直接 `postToHwRenderNode`，绕过 Channel/RS/协程
- `captureAndBlurFromSourceBitmap`：API 31+ 上截取 source 后直接 `postToHwRenderNode`，零协程
- 新增 `postToHwRenderNode()`：持久化 RenderNode，内容 + RenderEffect 模糊在 GPU 端合成
- 新增 `updateHwBlurRadius()`：blurRadius 变化时即时更新 RenderEffect
- 新增 `discardHwNode()`：尺寸变化时丢弃 HW 节点，dispatchDraw 自动回退 SW 路径
- SW 路径（API < 31）和 BlurStrategy / LatencyBenchmark 保持不变

## HomeActivity 双模式切换（Static / Video）
- 新增 **Static 模式**：使用 WallpaperManager 获取静态壁纸 Bitmap 作为 sourceBitmap
- 新增 **Video 模式**：布局中激活 MediaSurfaceView（alpha=0），设置 surfaceView 为动态模糊源
- 新增底部按钮 `btn_toggle_mode` 切换两种模式
- Video 模式管线：MediaPlayer → SurfaceView → PixelCopy 逐帧截取 → HW (RenderNode) 或 SW (RS) 模糊
- 两种模式的拖拽事件统一，根据 `isVideoMode` 标志调用 `captureAndBlur()` 或 `captureAndBlurFromSourceBitmap()`

## FrostedAnimatedGlowView 方案 A+B 延时优化
- **方案 A**：新增 `postToHwRenderNodeFromSource()`，sourceBitmap → RenderNode 直接绘制，跳过中间 capture Bitmap 分配+绘制。HW 路径 (API 31+) 每帧节约 0.3-0.8ms。
- **方案 B**：位置缓存 `mCachedScreenPos` + `onLayout` 中刷新 + `getScreenPos()` 优先返回缓存。`captureAndBlurFromSourceView` 和 `captureAndBlurFromSurfaceView` 均使用缓存。每帧节约 0.2-0.5ms。
- 移除旧代码中的多个 Log.d(TAG, ...) 调用。

## 分层模糊架构 — 尝试失败，回退至 capture 方案
- 尝试了 `FLAG_SHOW_WALLPAPER` + `Window.setBackgroundBlurRadius()` 系统级方案，用户设备不支持（黑屏）
- 已回退至确定性更高的 capture 方案（Static sourceBitmap + Video SurfaceView 双模式）
- 记录：FLAG_SHOW_WALLPAPER + window blur 在某些设备/ROM 上有兼容性问题，不可作为通用方案

- 重写 BoardWallpaperService，替换 FastBlurUtil 为双后端模糊引擎：

  - API < 31：RenderScript 硬件加速模糊（`blurAndDrawRs`）
- 抽取 `initBlurBackend()` / `releaseBlurBackend()` 统一管理模糊资源
- canvas.drawRenderNode 不支持时自动回退 RS 路径
- HomeActivity 新增 `btn_service_mode` 按钮 + `setServiceWallpaper()` 方法
- 服务级 blur 流程：用户点击按钮 → 系统壁纸选择器 → 设为动态壁纸 → Service 自动 blur 视频帧 → FLAG_SHOW_WALLPAPER 显示模糊背景
