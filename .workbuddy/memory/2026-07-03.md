## 视频播放 View 三方案

### 方案对比
| 特性 | CodecLoopVideoView | LoopVideoView | SurfaceLoopVideoView |
|------|-------------------|---------------|---------------------|
| 基类 | TextureView | VideoView | SurfaceView |
| 解码 | MediaCodec 硬解 | MediaPlayer | MediaCodec 硬解 |
| 渲染 | Surface(SurfaceTexture) | VideoView 内部 | SurfaceView Surface |
| 线程 | 独立解码线程 | VideoView 内部 | 独立解码线程 |
| 帧级控制 | 是 | 否 | 是 |
| 帧率控制 | targetFps | 无 | targetFps |
| 零拷贝 | 是 | 是 | 是 |

### CodecLoopVideoView 新增
- TextureView + MediaCodec + MediaExtractor 实现硬解码
- 独立后台解码线程，`dequeueInputBuffer` / `dequeueOutputBuffer` 循环
- 循环播放：EOS 时 `extractor.seekTo(0)` + `codec.flush()`
- 暂停：`shouldBePlaying` AtomicBoolean，暂停时 sleep 50ms 但不释放 codec
- `BUFFER_FLAG_CODEC_CONFIG` 标记 flush 完成后的首个空 buffer

## LoopVideoView 创建与增强

### 初版创建
- 新建 `core/core-ui/src/main/java/com/cn/core/ui/view/media/LoopVideoView.kt`
- 继承 SurfaceView + SurfaceHolder.Callback
- 核心特性：固定视频路径 + 循环播放 + 可见性感知（VISIBLE 播放/GONE 暂停）
- 生命周期管理：onAttachedToWindow 准备，onDetachedFromWindow 释放 MediaPlayer
- 默认静音播放（isMute=true），避免干扰

### 壁纸感知增强
- 新增 `wallpaperAware` 属性：启用后监听系统壁纸变化
- 动态壁纸检测：`WallpaperManager.getInstance(context).wallpaperInfo != null`
- 壁纸变化监听：`WallpaperManager.OnColorsChangedListener`（API 27+）
- 行为：动态壁纸 → 自动 VISIBLE + play；静态壁纸 → 自动 GONE + pause
- 生命周期集成：attach 时重新注册 + 初始状态检查，detach 时卸载

### 排查日志添加
- 全链路 Log.d/w/e 覆盖 TAG="LoopVideoView"，`adb logcat -s LoopVideoView` 过滤
- 覆盖点：videoPath 设置、surfaceCreated/Destroyed、onWindowVisibilityChanged、onVisibilityChanged、handleVisibilityChange、play/pause、resumePlayback、prepareVideo、onPrepared/onCompletion/onError、startMediaPlayer、pauseMediaPlayer、releaseMediaPlayer、onAttach/Detach、壁纸感知全流程
- 关键决策点：为何不 prepare（path empty / surface not ready）、为何不 start（mp null / not prepared / already playing）、onPrepared 是否触发 start、isLiveWallpaper 检测结果、applyWallpaperState visibility 切换

### 壁纸监听移除 + VideoView 重写
- 用户要求去除壁纸感知，改为纯显示即播放
- 随后重写为 VideoView 子类（485行 → 181行）
- VideoView 内置 MediaPlayer/SurfaceHolder 管理，无需手动实现 Callback
- 循环播放：`OnCompletionListener` 内 `seekTo(0)` + `start()`
- 保持原有：videoPath 属性、isLooping/isMute、可见性感知、XML 属性、调试日志

## BlurView 全屏模糊修复（方案 F）

### 根因
BlurView SYSTEM 路径 `onSystemBlurReady()` 无条件调用 `setupWindowBlur()` 设置 Window 级 `FLAG_BLUR_BEHIND`。
在 decorView 透明且无 BlurMaskManager 蒙版遮挡的界面上，全窗口模糊泄漏。

### 第二轮修复：堵 enablePipeline 漏洞
- **BlurView.kt L291**：`enablePipeline` 同样加 `BlurMaskManager.hasMask` 守卫
- 改动前 `enablePipeline = useHiddenBlurPipeline || preSetBlurBehind` → 有 FAGV 的界面即使无蒙版也注入 `Window.setBackgroundBlurRadius`（机制②全窗口模糊）
- 改动后 `enablePipeline = if (hasMask) (useHiddenBlurPipeline || preSetBlurBehind) else false` → 无蒙版时跳过机制②

### 第三轮修复：P0-1 + P0-2 + P1（3 个文件）

**P0-1: deactivateBlur 无清理机制 B**
- BackgroundBlurDrawable.kt：新增 `clearWindowBackgroundBlurRadius(window)` 公开方法
- BlurView.kt：新增 `selfSetBackgroundBlurRadius` 字段，activateBlur 时 `= enablePipeline` 记录，deactivateBlur SYSTEM 分支调用 `clearWindowBackgroundBlurRadius` 清零
- 修复前：Window.setBackgroundBlurRadius(25) 激活时设了，deactivateBlur 只清机制 A(FLAG_BLUR_BEHIND)，机制 B 残留 → 全屏模糊

**P0-2: BlurMaskManager.unregister 蒙版消失后窗口模糊裸奔**
- BlurMaskManager.kt unregister()：refCount=0 移除蒙版前，先清 FLAG_BLUR_BEHIND + blurBehindRadius=0 + clearWindowBackgroundBlurRadius
- 修复前：FAGV detach → 蒙版移除 → 但 FLAG_BLUR_BEHIND + setBackgroundBlurRadius 仍在 → BlurView 仍 active → 全屏模糊

**P1: SYSTEM 路径补 isCrossWindowBlurEnabled 前置检查**
- BlurView.kt activateBlur()：SYSTEM 分支前检查 `BlurMaskManager.isCrossWindowBlurEnabled(act)`，OEM 禁用时跳过 SYSTEM 直接降级 FALLBACK

### 第四轮：添加全链路诊断日志
- **BlurView.kt**：activateBlur / onSystemBlurReady / setupWindowBlur / deactivateBlur 全路径加 logD/logW
  - activateBlur：输出 radius/forceFallback/bgBlurOnly/hasMask/crossWindowBlur/enablePipeline/preSetBlurBehind + 最终路径选择
  - onSystemBlurReady：输出 hasMask/bgBlurOnly/selfSetBgBlurR + 分支选择(clearExternal/setupWindow/skip)
  - setupWindowBlur：输出 before/after 的 flags(hex)+blurR + skip 原因
  - deactivateBlur：输出 mode/selfSetWindowBlur/selfSetBgBlurR + 清理 A/B/C 各步骤
- **BackgroundBlurDrawable.kt**：
  - tryCreateSystem 机制 B 调用时改 logW 级别 + `>>> ... CALLED` 前缀，突出窗口级模糊注入
  - clearWindowBackgroundBlurRadius 输出 OK/null/异常
- **BlurMaskManager.kt**：
  - register：输出 NEW vs existing + refCount + act 类名
  - unregister：输出 refCount + refCount=0 时 hadFlag/blurR + 清理步骤 + 蒙版移除确认
- **日志过滤命令**：`adb logcat -s BlurView:* BgBlurDrawable:* BlurMaskMgr:* BgBlurHelper:*`

### 第六轮：无蒙版时彻底跳过 SYSTEM 路径（最终修复）
- **根因（日志确认）**：`hasRegions=true` 但仍全屏模糊 — BackgroundBlurDrawable 是窗口级 Drawable，模糊作用在窗口背景层，无 WallpaperMaskDrawable 遮挡时整层模糊透出
- **关键发现**：之前项目记忆"③ 能独立工作，不依赖 ①②"在当前 ROM 不成立 — BackgroundBlurDrawable 无蒙版时也会全屏泄漏
- **修复**：`activateBlur()` 中 `hasMask=false` 时直接跳过 SYSTEM 降级 FALLBACK，不再尝试 BackgroundBlurDrawable
  - 同时清除可能存在的 orphaned FLAG_BLUR_BEHIND（`preSetBlurBehind=true` 时 `clearExternalBlurBehind()` + `clearedOrphanedBlur=true`）
- **路径选择逻辑（最终版）**：
  ```
  hasMask=true + crossWindowBlur=true → SYSTEM (BackgroundBlurDrawable + FLAG_BLUR_BEHIND，蒙版遮挡安全)
  hasMask=false                      → FALLBACK (截屏+RenderEffect per-View，安全)
  crossWindowBlur=false              → FALLBACK (OEM 禁用)
  forceFallback=true                 → FALLBACK
  ```
- **BlurMaskManager.kt L85**：新增 `hasMask(activity)` 查询方法，线程安全检查 masks.containsKey
- **BlurView.kt L304**：`onSystemBlurReady()` 中仅在 `BlurMaskManager.hasMask == true` 时调用 `setupWindowBlur()`，
  无蒙版时跳过 FLAG_BLUR_BEHIND，仅依赖 BackgroundBlurDrawable per-View 模糊（已验证可独立工作）

### 逻辑
```
有 WallpaperBlurView 的界面 → BlurMaskManager 已注册 → setupWindowBlur (FLAG_BLUR_BEHIND) → 蒙版遮挡非 View 区 ✅
无 WallpaperBlurView 的界面 → BlurMaskManager 未注册 → 跳过 FLAG_BLUR_BEHIND → 仅 BackgroundBlurDrawable per-View ✅
```
