## WallpaperBlurView V2 (Compositor Blur) 重写
- 基于 Android 12+ 系统权限，利用 `blurBehindRadius` 让 SurfaceFlinger GPU 合成器原生模糊
- 核心优化：零CPU/零内存/零延迟，120fps+ 天然支持动态壁纸

### V2 架构
- **Compositor Blur 主路径**: `WindowManager.LayoutParams.blurBehindRadius` → SurfaceFlinger GPU 高斯模糊
- **蒙版反切 (WallpaperMaskDrawable)**: 窗口背景绘制清晰壁纸 + clipPath EVEN_ODD 镂空 View 区域 → 仅 View 区域显示模糊壁纸

- **移除**: RenderScript / Bitmap 池 / ScreenLayerCapture 反射 / Choreographer 帧回调(主路径) / ultraLowLatency 模式
- 视觉效果保留：圆角/描边/蒙层/内发光/顶部高光/5状态动画

### 关键技术
- `blurBehindRadius` 通过反射设置（@hide 字段，系统权限下可访问）
- `CrossWindowBlurListeners` 公开API 检测跨窗口模糊支持
- `WallpaperMaskDrawable` 按 Activity 级共享，引用计数管理生命周期
- attrs.xml 移除 `wbv_blurScaleFactor` / `wbv_ultraLowLatency`（V2 不需要）

## WallpaperBlurView V2.5 修复（模糊不生效）
- **问题根因**：缺少 `FLAG_BLUR_BEHIND` 标志位 + 使用反射访问公开 API + 未用 `WindowManager.isCrossWindowBlurEnabled` 检测
- **修复方案**：
  1. 反射 → 公开 API：`params.setBlurBehindRadius(radiusPx)` + `params.blurBehindRadius`（API 31 公开方法/属性）
  2. 添加 `FLAG_BLUR_BEHIND`：启用 blur behind 的必要标志，之前缺失导致模糊不生效
  3. `WindowManager.isCrossWindowBlurEnabled`：公开 API（非 @hide），检测 OEM 是否启用跨窗口模糊
  4. 删除全部反射代码：`blurRadiusField` / `getDeclaredField("blurBehindRadius")` / `isBlurBehindRadiusAvailable()`
  5. `deactivateCompositorBlur` 清理时同时移除 `FLAG_BLUR_BEHIND`
  6. Fallback 保留：`WallpaperManager.getDrawable()` → RenderNode + RenderEffect GPU 模糊
- **关键事实修正**：`blurBehindRadius` / `setBlurBehindRadius()` / `FLAG_BLUR_BEHIND` 都是 API 31 (Android 12) 公开 API，非 @hide
- **编译验证**：`./gradlew :core:core-ui:compileDebugKotlin` BUILD SUCCESSFUL

## WallpaperBlurView V2 黑屏诊断修复
- **现象**：实际效果 View 区域显示为黑色/深色，而非模糊壁纸
- **诊断出 3 个根因，全部修复**：

### 修复 1: 坐标系错误（最致命）
- `updateMaskHole()` 使用 `getLocationOnScreen()` 获取屏幕坐标（含状态栏高度）
- 但 WallpaperMaskDrawable 绘制在 decorView 画布上，坐标系从窗口左上角开始
- **修复**：获取 `decorView.getLocationOnScreen()` 做偏移转换 screen→window 坐标

### 修复 2: `getOpacity()` 返回 OPAQUE
- WallpaperMaskDrawable 报告 OPAQUE → 系统认为窗口背景不透明 → 阻断 compositor blur 可见性
- **修复**：改为 `PixelFormat.TRANSLUCENT`

### 修复 3: LAYER_TYPE_HARDWARE 导致 View 不透明
- `init` 中设置 `LAYER_TYPE_HARDWARE` 会创建不透明硬件层 buffer，遮挡模糊壁纸
- **修复**：
  - init 改为 `LAYER_TYPE_NONE`
  - `activateCompositorBlur()` 设置 `LAYER_TYPE_NONE`（View 必须透明）
  - `activateWallpaperBitmapBlur()` 设置 `LAYER_TYPE_HARDWARE`（drawRenderNode 需要）

### 修复 4: 壁纸绘制映射
- WallpaperMaskDrawable 原来用 `getBounds()` 绘制壁纸 bitmap，可能不对齐画布尺寸
- **修复**：使用 `canvas.width/height` 作为 dstRect，确保壁纸铺满整个 decorView 画布

### 调试日志增强
- WallpaperMaskDrawable.draw() 增加 canvas/bmp/holes 日志
- 每个 hole 的坐标位置日志

## WallpaperBlurView 移动跟随修复
- **问题**：View 移动（拖拽/滚动/动画）时模糊不跟随
- **根因**：
  1. 缺少 `ViewTreeObserver.OnScrollChangedListener` — 父容器滚动不触发
  2. 缺少 `OnGlobalLayoutListener` — 布局变化导致位置改变不触发
  3. `updateMaskHole()` 后未 `invalidate` decorView — `invalidateSelf()` 不够
  4. 缺少 `setX/setY`/`offsetTopAndBottom`/`offsetLeftAndRight` override
- **修复**：
  - 新增 `onPositionChanged()` 统一位置变化回调
  - 新增 `installPositionTracking()`/`uninstallPositionTracking()` 管理 ViewTreeObserver 监听
  - `updateMaskHole()` 末尾加 `decorView.invalidate()`
  - 补全 7 个位置相关 override：setTranslationX/Y, setX/Y, offsetTopAndBottom, offsetLeftAndRight, scrollBy
  - 所有位置回调统一走 `onPositionChanged()`
- **编译验证**：BUILD SUCCESSFUL

## WallpaperBlurView 静态壁纸对齐修复
- **问题**：动态壁纸正常，静态壁纸不对
- **根因**：`WallpaperMaskDrawable` 中 `drawable.toBitmap()` 将 WallpaperManager 返回的 Drawable
  转为原始 Bitmap，然后拉伸到 `canvasW×canvasH`。但系统显示壁纸用 CENTER_CROP 逻辑，
  壁纸 Bitmap 可能远大于屏幕（如 2880×2560 的滚动壁纸），拉伸后严重偏离系统实际显示
- **动态壁纸为何正常**：Live Wallpaper 时 `WallpaperManager.getDrawable()` 返回 null →
  mask 不绘制任何内容 → 整个窗口透明 → Compositor Blur 模糊壁纸全局可见
- **修复**：WallpaperMaskDrawable 保留 Drawable 引用而非转 Bitmap
  - `wallpaperBitmap: Bitmap?` → `wallpaperDrawable: Drawable?`
  - `drawable.toBitmap()` → `drawable.mutate()` 保留引用
  - `canvas.drawBitmap(bmp, srcRect, dstRect, paint)` → `wd.draw(canvas)`
  - 设置 `wd.setBounds(0, 0, canvasW, canvasH)` 后由 Drawable 自身处理缩放对齐
  - 移除 `bgPaint` / `srcRect` / `dstRect` 字段（不再需要）
- **编译验证**：BUILD SUCCESSFUL

## WallpaperBlurView 壁纸切换跟随

  - `WallpaperMaskDrawable` 新增 `markWallpaperDirty()` 方法

## WallpaperBlurView 动态壁纸黑屏修复
- **现象**：静态壁纸显示/镜像均正常；动态壁纸显示屏黑，镜像正常
- **根因**：动态壁纸时 `WallpaperManager.getDrawable()` 返回 null →
  `WallpaperMaskDrawable.draw()` 直接 return → 但 decorView 仍持有该 Drawable 作为 background →
  系统为窗口分配了不透明 buffer → 阻断了 Compositor Blur 的透明通道 → 黑屏
- **镜像为何正常**：镜像走不同的合成管线，可能绕过了这个限制
- **修复**：
  - 新增 `isLiveWallpaper()` 方法，检测 `WallpaperManager.wallpaperInfo != null`
  - 动态壁纸时跳过 `registerWithMask()` — 不设置 WallpaperMaskDrawable 为 decorView background
  - 动态壁纸时 decorView background 保持 null（完全透明），Compositor Blur 直接模糊动态壁纸
  - `updateMaskHole()` 动态壁纸时跳过（无蒙版需要更新）
  - 仍保留 `FLAG_BLUR_BEHIND` + `blurBehindRadius`（可能本身是 OK 的，是 mask 阻挡了）
- **已知限制**：动态壁纸下整个窗口透明 → 区域外也显示模糊壁纸（无清晰壁纸反切）
- **编译验证**：BUILD SUCCESSFUL

## WallpaperBlurView 动态壁纸切换后不显示修复
- **现象**：设置动态壁纸后，使用该 View 的 Activity 不再显示动态壁纸
- **根因**：静态→动态壁纸切换时，`onWallpaperChanged()` 只做了数据刷新（markWallpaperDirty），没有重新选择模糊路径：
  1. `WallpaperMaskDrawable` 仍在 decorView 上，但 `getDrawable()` 返回 null → `draw()` 什么都不画
  2. decorView 有一个空 Drawable 背景 → 窗口不透明/黑 → 动态壁纸不可见
- **修复**：`onWallpaperChanged()` 中检测壁纸类型变化，调用 `deactivateBlur()` + `activateCompositorBlur()` 重新选择路径：
  - 静态→动态：移除 WallpaperMaskDrawable，恢复原始背景，走动态壁纸分支（跳过 mask）
  - 动态→静态：注册 WallpaperMaskDrawable，绘制清晰壁纸+镂空
- **编译验证**：待验证

## WallpaperBlurView 仅 View 区域模糊修复（动态壁纸）
- **现象**：动态壁纸时整个窗口都显示模糊效果，而非仅 View rect 区域
- **根因**：`FLAG_BLUR_BEHIND` 让 SurfaceFlinger 对整个壁纸层做高斯模糊，无法限定区域
  - 静态壁纸用 `WallpaperMaskDrawable` 画清晰壁纸覆盖模糊区域 → 仅 View 区域模糊 ✅
  - 动态壁纸 `getDrawable()` 返回 null，无法画蒙版 → 整个窗口模糊 ❌
- **修复**：动态壁纸不再使用 Compositor Blur，改用 Bitmap 模糊路径
  - `activateCompositorBlur()` 动态壁纸分支：调用 `activateWallpaperBitmapBlur()` 替代设置 `FLAG_BLUR_BEHIND`
  - `updateWallpaperBlurNode()` 增加 `WallpaperInfo.loadThumbnail()` 作为动态壁纸模糊源
  - 效果：View 内部绘制模糊壁纸，窗口其他区域透明 → 清晰壁纸自然透出
- **编译验证**：BUILD SUCCESSFUL

## WallpaperBlurView 动态壁纸模糊内容显示静态壁纸修复
- **现象**：动态壁纸时，View 区域模糊内容是静态壁纸而非动态壁纸缩略图
- **根因**：`updateWallpaperBlurNode()` 先调 `wm.drawable`，判断 `drawable != null` 才加载。
  但某些设备/ROM 上，设置了动态壁纸后 `WallpaperManager.getDrawable()` 仍返回非 null（旧的静态壁纸缓存），
  导致代码走到 `drawable != null` 分支，直接用了静态壁纸 bitmap，根本不会走 `else` 分支去 `loadThumbnail()`
- **修复**：`updateWallpaperBlurNode()` 中先调用 `isLiveWallpaper()` 判断壁纸类型，再选择数据源：
  - 动态壁纸 → 直接走 `WallpaperInfo.loadThumbnail()`，跳过 `getDrawable()`
  - 静态壁纸 → 使用 `WallpaperManager.getDrawable()`
- **编译验证**：BUILD SUCCESSFUL

## WallpaperBlurView 动态壁纸缩略图内容错误修复
- **现象**：动态壁纸时 View 区域显示绿色网格 + Android 机器人图标（`loadThumbnail()` 的预览缩略图），跟实际动态壁纸内容完全不符
- **根因**：`WallpaperInfo.loadThumbnail()` 返回的是动态壁纸开发者在 manifest 中注册的**静态预览图**，跟实际渲染内容无关。Android 没有标准 API 让外部应用获取动态壁纸的实时帧
- **修复**：使用系统权限通过反射调用 `SurfaceControl.screenshot()` 直接捕获屏幕内容
  - 新增 `captureScreenRegion()`：反射调用 `SurfaceControl.screenshot(IBinder, Rect, int, int, boolean, int)` 捕获屏幕特定区域
  - 新增 `scheduleScreenCapture()`：调度帧同步捕获流程
    1. `isCapturingScreen=true` → `dispatchDraw()` 跳过模糊绘制 → View 透明
    2. 下一帧 Choreographer 回调执行 `screenshot()` → 捕获到壁纸（View 透明）
    3. 恢复 `isCapturingScreen=false` → 用捕获到的 bitmap 更新模糊内容
  - `updateWallpaperBlurNode()` 动态壁纸分支：优先使用 `screenCaptureBitmap`，无则调度捕获
  - `onPositionChanged()` 动态壁纸路径：移动时调度 `scheduleScreenCapture()`
  - `deactivateBlur()`：清理 `screenCaptureBitmap`
  - `dispatchDraw()`：`isCapturingScreen` 为 true 时跳过模糊绘制
- **要求**：需要系统权限 (`READ_FRAME_BUFFER`)，反射调用 `@hide` API
- **编译验证**：BUILD SUCCESSFUL
